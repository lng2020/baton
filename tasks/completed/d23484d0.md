# Add `baton init` CLI command for project creation from GitHub repo

Create a new CLI entry point `baton-init` that interactively creates a project from a GitHub repo URL.

## What to implement

Add a new file `backend/init_project.py` with a `main()` function that:

### 1. CLI Arguments (argparse)
- `repo` (positional, required) — GitHub repo URL, e.g. `https://github.com/user/repo` or shorthand `user/repo`
- `--path` (optional) — Local clone destination. Default: `~/projects/<repo_name>`
- `--name` (optional) — Display name. Default: derived from repo name (capitalize)
- `--id` (optional) — Project ID. Default: derived from repo name (lowercase, hyphens to underscores)
- `--color` (optional) — Hex color. Default: `#0f3460`
- `--agent-port` (optional, int) — Agent port. Default: auto-assign starting from 9100, incrementing to avoid conflicts with existing projects in projects.yaml
- `--no-clone` (optional, flag) — Skip cloning if repo already exists locally

### 2. Interactive flow (with prompts for confirmation)
The command should:

1. **Parse the GitHub repo URL**: normalize `user/repo` to `https://github.com/user/repo`. Extract owner and repo name.
2. **Determine local path**: Use `--path` or default `~/projects/<repo_name>`. Ask user to confirm: `Clone to /path/to/repo? [Y/n]`
3. **Clone the repo**: Run `git clone <url> <path>` via `subprocess.run`. Skip if `--no-clone` or path already exists (with a warning).
4. **Determine project ID, name, description**: Auto-derive from repo name but let user override interactively:
   - `Project ID [repo_name]:` (press Enter to accept default)
   - `Project name [Repo Name]:` (press Enter to accept default)
   - `Description []:` (press Enter to skip)
5. **Auto-assign agent port**: Read existing `config/projects.yaml`, find max port in use, assign next available. Show: `Agent URL: http://localhost:9101`
6. **Create tasks directory structure**: Create `<path>/tasks/{pending,in_progress,completed,failed}/` directories.
7. **Create agent.yaml** in the project root if it doesn't exist — copy from baton's own `agent.yaml` as template.
8. **Update `config/projects.yaml`**: Load existing config, append new project entry, write back. Use the `yaml` library. The new entry should include all fields: `id`, `name`, `path`, `repo` (the GitHub URL), `description`, `color`, `tasks_dir: tasks`, `agent_url`.
9. **Print summary**: Show the created project config and next steps:
   ```
   Project 'repo_name' created!
   - Path: /path/to/repo
   - Agent URL: http://localhost:9101
   
   Next steps:
     cd /path/to/repo && baton-agent --port 9101
   ```

### 3. Register the CLI entry point

In `pyproject.toml`, add to `[project.scripts]`:
```
baton-init = "backend.init_project:main"
```

### 4. Error handling
- If `git clone` fails, print error and exit
- If project ID already exists in `config/projects.yaml`, warn and exit
- If target path exists and `--no-clone` is not set, ask user if they want to use existing directory
- Handle missing `config/projects.yaml` by creating it with empty projects list

### 5. Key implementation details
- Use `pathlib.Path` for all path operations
- Use `yaml.safe_load` / `yaml.dump` for YAML operations (with `default_flow_style=False`)
- Use `input()` for interactive prompts
- The config path should be resolved relative to the baton installation: `Path(__file__).parent.parent / 'config' / 'projects.yaml'` (same as `backend/config.py`)
- Import and reuse `load_config` from `backend.config` where possible

### Files to modify:
- **CREATE**: `backend/init_project.py` — main implementation
- **EDIT**: `pyproject.toml` — add `baton-init` script entry

```bash
pip install -e .

```
